# 贡献说明

谢谢你愿意花时间改 AnyPeek。

这个项目欢迎的，不只是“写大功能”。修一个小 bug、把文档写清楚、补一类预览、让某个交互少绕一步，都算很有价值的贡献。

这页主要回答三件事：

- 什么时候可以直接提 PR
- 什么时候最好先讨论一下
- 改完以后，最少要自查到什么程度

## 开工前先看哪几页

不用把整个仓库从头读到尾，按改动类型选就行：

- 想先知道项目定位：看 [README](../README.md)
- 需要把项目跑起来：看 [部署指南](./DEPLOYMENT.md)
- 要动预览链路、代理、安全边界或大文件逻辑：看 [技术细节](./TECHNICAL_DETAILS.md)

## 哪些改动可以直接提 PR

这些通常可以直接开工：

- 文档润色和错别字修正
- 小的样式或交互优化
- 范围明确的 bug fix
- 不改变项目定位的小功能补齐

如果你准备动的是下面这些，最好先开个 issue 或讨论一下：

- 新增一整类预览能力
- 修改代理逻辑或安全限制
- 调整部署方式、运行时入口或平台支持范围
- 会明显改变 README 主叙事、产品定位或使用边界的改动

这样做不是为了设门槛，而是为了避免你花很多时间，最后发现方向根本没对上。

## 本地开发最常用的命令

```bash
pnpm install --frozen-lockfile
pnpm dev
```

如果你要在提交前做一轮完整检查，常用命令是：

```bash
pnpm test
pnpm run check:cf
pnpm run check:deno
pnpm run build
```

## 提交 PR 时，什么样的说明最省沟通

一个容易 review 的 PR，通常会把这几件事说清楚：

- **改了什么**：一句话概括改动范围
- **为什么改**：是修 bug、补体验，还是补支持范围
- **用户会感受到什么变化**：尤其是交互或预览行为变了的时候
- **你自己验证了什么**：跑了哪些命令、测了哪些样例

如果是界面改动，带一张截图或一段录屏，通常会省很多来回确认。

## 自查不用“一刀切”

不是每次都要把整套检查跑满，按改动类型来就够了。

### 只改文档

- 自己完整通读一遍
- 确认相对链接可点
- 确认表述和仓库现状一致

### 改前端交互或预览逻辑

至少跑：

```bash
pnpm test
pnpm run build
```

### 改 Cloudflare / Deno / Vercel / 代理相关逻辑

建议再补：

```bash
pnpm run check:cf
pnpm run check:deno
```

### 改桌面版相关逻辑

如果你手头有 Windows 环境，最好再跑：

```bash
pnpm exec tauri info
```

## 什么时候要顺手更新文档

如果你的改动影响了对外认知、部署步骤或技术边界，最好一起把文档补上。

大致按这个分工更新就够了：

- `README.md`：项目是什么、适合谁、支持什么、从哪里开始
- `docs/DEPLOYMENT.md`：本地和各平台怎么部署
- `docs/TECHNICAL_DETAILS.md`：实现思路、主链路、安全和大文件策略
- `docs/MAINTAINERS.md`：仓库维护、自动部署、长期整理

## 一个简单但很重要的约定

AnyPeek 的主线一直是：**把 `URL / 本地文件 -> 可读内容` 这条路径做顺。**

所以改代码时，尽量优先保持这几件事：

- URL 和本地文件两条路径的体验不要越改越分裂
- 安全边界变了时，要把原因写清楚
- 大文件场景不要因为“支持更多格式”而顺手被拖慢
- 改动尽量聚焦，不把无关重构一起塞进来
